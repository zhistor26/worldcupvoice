# 懒猫微服开发者手册

> **本仓库权威手册正文**（`docs/lazycat/开发者手册.md`）。人类可通读；AI 执行任务时**优先**读 `.cursor/skills/lazycat-porting/SKILL.md`，再按 `porting-handbook.md` 章节索引**按需打开**本文对应 §，避免与 skills 重复全文加载。

本文根据已提供的懒猫微服开发者手册材料整理，定位为通用开发手册。它不绑定具体业务仓库实现，主要用于指导懒猫微服应用的开发、调试、打包、发布和移植。

说明：

- 原始材料来自网页拷贝，已去掉导航、分页、重复片段和明显网页噪音。
- 原始材料中 `数据库服务`、`Redis`、`deploy-params` 片段存在截断，本文只整理可确认内容。
- 命令、路径、配置字段名保持原文。

## 1. 总体认知

懒猫微服应用面向用户侧通常表现为一个通过 HTTPS 访问的 Web App。应用最终交付格式是 `.lpk`，代码、静态资源、运行声明、镜像信息等会被打进 LPK 后安装到目标微服运行。

同一份 LPK 部署后，可在 Android、iOS、macOS、Windows、Web 浏览器等多端访问，具体平台支持也可以通过配置限制。

核心概念：

- `lzc-cli`：开发、构建、部署、发布、查看日志、进入容器的命令行工具。
- `lzc-build.yml`：构建配置，也是发布配置。
- `lzc-build.dev.yml`：开发态覆盖配置，`project deploy`、`project info`、`project exec` 等命令默认优先使用它。
- `lzc-manifest.yml`：应用运行结构配置，描述 `application`、`routes`、`services` 等。
- `package.yml`：应用静态元数据，LPK v2 新项目应把 `package`、`version`、`name`、`description` 等静态字段放在这里。
- `/lzcapp/var`：应用持久化数据目录，重启后保留。
- `/lzcapp/pkg/content`：LPK 打包内容在运行时的只读目录。

## 2. 开发环境搭建

目标：本地机器可以使用 `lzc-cli` 连接目标微服，并具备构建与部署能力。

前置条件：

- 有一台可访问的懒猫微服。
- 账号可以登录懒猫微服客户端。
- 终端可执行 `node` 和 `npm`。

步骤：

1. 安装 Node.js 18+，建议使用 LTS 版本。
2. 安装并登录懒猫微服客户端。
3. 在微服应用商店安装“懒猫开发者工具”。
4. 安装 `lzc-cli`：

```bash
npm install -g @lazycatcloud/lzc-cli
lzc-cli --version
```

如果 `lzc-cli --version` 能输出版本号，说明安装成功。

首次使用时准备 SSH Key：

```bash
[ -f ~/.ssh/id_ed25519.pub ] || ssh-keygen -t ed25519 -f ~/.ssh/id_ed25519 -N ""
```

选择默认目标微服：

```bash
lzc-cli box list
lzc-cli box switch <boxname>
lzc-cli box default
```

如果在 WSL 或 LightOS 等无法直接使用 hclient API 的环境中开发，可用 SSH 方式添加目标微服：

```bash
lzc-cli box add-by-ssh root <微服局域网IP>
```

注意：`<微服局域网IP>` 必须填写微服局域网 IP，不要填写域名。该方式要求目标微服已开通 SSH。

仅当通过 hclient 模式接入目标微服时，才需要上传公钥：

```bash
lzc-cli box add-public-key
```

`add-by-ssh` 模式不需要、也无法成功执行 `lzc-cli box add-public-key`。

验证：

```bash
lzc-cli --version
lzc-cli box default
lzc-cli project --help
```

通过标准：

- `lzc-cli` 版本号可见。
- `box default` 能输出默认微服名。
- `project --help` 能看到 `deploy`、`start`、`info`、`exec`、`cp`、`log`、`release` 等命令。

常见问题：

- `No default box configured`：执行 `lzc-cli box list` 和 `lzc-cli box switch <boxname>`。
- `permission denied (publickey)`：hclient 模式执行 `lzc-cli box add-public-key`；`add-by-ssh` 模式检查目标微服 SSH 公钥授权配置。
- `lzc-cli: command not found`：确认 npm 全局安装路径已加入 `PATH`，或重新打开终端。

## 3. Hello World 与开发态验证

创建模板项目：

```bash
lzc-cli project create hello-lpk -t hello-vue
cd hello-lpk
```

模板默认会生成：

- `lzc-build.yml`：默认构建配置，也是发布配置。
- `lzc-build.dev.yml`：开发态覆盖配置，通常包含独立 dev 包名、空 `contentdir` 覆盖和 `DEV_MODE=1`。

部署并查看入口：

```bash
lzc-cli project deploy
lzc-cli project info
```

说明：

- 首次部署如果出现授权提示，按 CLI 输出打开浏览器完成授权。
- `project` 命令默认优先使用 `lzc-build.dev.yml`。
- 命令输出里的 `Build config` 表示本次实际命中的构建配置文件。
- 如需操作发布配置，显式加 `--release`。
- `project info` 在应用已运行时会输出 `Target URL`。

打开应用：

- Android / iOS：在懒猫微服客户端启动器中点击应用图标。
- macOS / Windows：在桌面客户端启动器中点击应用图标。
- Web：复制 `project info` 输出的 `Target URL` 到浏览器访问。

启动前端 dev server：

```bash
npm run dev
```

对于 `hello-vue` 模板，首次打开可能先看到前端开发引导页。它表示入口已进入 request inject 分流脚本控制，并正在等待本地前端 dev server。

修改 `src/App.vue` 后刷新页面或等待热更新，即可验证开发链路。

排查日志：

```bash
lzc-cli project log -f
```

查看 LPK 交付包：

```bash
lzc-cli project release -o hello.lpk
lzc-cli lpk info hello.lpk
```

验证标准：

- `lzc-cli project info` 显示 `Current version deployed: yes`。
- `lzc-cli project info` 显示 `Project app is running.`。
- 应用入口可打开。
- 启动 `npm run dev` 后浏览器和客户端都能进入前端页面。
- 修改前端源码后能看到新内容。

常见问题：

- `Project app is not running. Run "lzc-cli project start" first.`：执行 `lzc-cli project start`。
- `Target URL` 未显示：执行 `lzc-cli project start` 后再执行 `lzc-cli project info`。
- 页面提示前端 dev server 未就绪：确认页面提示端口，在项目目录执行 `npm run dev` 后刷新。

## 4. HTTP 路由与后端对接

目标：区分 `exec://`、`http://`、默认鉴权、`application.public_path` 和持久化目录。

创建带后端模板：

```bash
lzc-cli project create hello-api -t todolist-golang
cd hello-api
```

先部署默认 `exec` handler：

```bash
lzc-cli project deploy
lzc-cli project info
```

继续前确认 `lzc-cli project info` 输出 `Project app is running.`。如果还在启动中，等待几秒后再次执行。

在已登录浏览器的开发者工具 Console 中验证：

```javascript
fetch('/api/health').then((r) => r.json()).then(console.log);
fetch('/api/todos').then((r) => r.json()).then(console.log);
```

期望：

- `/api/health` 返回 `{"status":"ok"}`。
- `/api/todos` 返回 `items` 列表，初次通常为空。

模板默认路由示例：

```yml
application:
  routes:
    - /=exec://3000,/app/run.sh
```

`exec://3000,/app/run.sh` 的含义是执行 `/app/run.sh`，并把流量转发到 `127.0.0.1:3000`。它适合单容器后端快速起步。

新增第三方 service 演示 `http` handler：

```yml
application:
  image: embed:app-runtime
  routes:
    - /inspect/=http://whoami:80/
    - /=exec://3000,/app/run.sh

services:
  whoami:
    image: registry.lazycat.cloud/traefik/whoami
```

说明：

- `/inspect/` 走 `http://whoami:80/`，不会触发 `/app/run.sh`。
- `http://` handler 只负责转发，不负责启动进程。
- `/inspect/` 要放在 `"/="` 之前，避免被更宽泛规则先匹配。

部署并验证：

```bash
lzc-cli project deploy
lzc-cli project info
```

浏览器 Console：

```javascript
fetch('/inspect/').then((r) => r.text()).then(console.log);
```

期望输出中可见 `whoami` 返回的请求信息，例如 `Hostname`、`RemoteAddr`、`Headers`。

默认情况下，微服应用 HTTPS 路径受登录态保护。未登录 `curl` 通常会被拦截，这是预期行为。如需放开健康检查：

```yml
application:
  public_path:
    - /api/health
```

重新部署后验证：

```bash
lzc-cli project deploy
lzc-cli project info
curl "<Target URL>/api/health"
```

`public_path` 只应放开必要路径，避免过度暴露接口。

查看持久化目录：

```bash
lzc-cli project exec -s app /bin/sh
ls -la /lzcapp/var
```

后端数据文件应放在 `/lzcapp/var` 下，例如 `/lzcapp/var/todos.json`。只有 `/lzcapp/var/` 目录下的内容会在重启应用后保留。

常见问题：

- `/inspect/` 返回 `404` 或 `502`：检查路由、顺序、`services.whoami.image`，并执行 `lzc-cli project log -s whoami -f`。
- `curl <Target URL>/api/health` 被拦截或跳转登录：使用已登录浏览器 `fetch` 验证，或按需配置 `application.public_path`。
- 部署后结果没变化：重新执行 `lzc-cli project deploy` 和 `lzc-cli project info`，确认 `Current version deployed: yes`。

## 5. 路由规则

`application.routes` 字段为 `[]Rule` 类型，按照 `URL_PATH=UPSTREAM` 形式声明：

- `URL_PATH`：浏览器访问时的实际 URL 路径，不含 hostname。
- `UPSTREAM`：具体上游服务。

支持的上游协议：

- `file:///$dir_path`
- `exec://$port,$exec_file_path`
- `http(s)://$hostname/$path`

默认前缀裁剪规则：

```yml
routes:
  - /api/=http://backend:80
```

当浏览器请求 `/api/v1` 时，后端实际收到 `/v1`。如果需要保留前缀，使用 `application.upstreams` 并设置 `disable_trim_location: true`，要求 `lzcos v1.3.9+`。

### 5.1 http 上游

`http/https` 支持内网或外网服务：

```yml
routes:
  - /=https://appstore.lazycat.cloud
```

应用内 service 示例：

```yml
application:
  routes:
    - /=http://bitwarden.cloud.lazycat.app.bitwarden.lzcapp:80
    - /or_use_this_short_domain=http://bitwarden:80
  subdomain: bitwarden

services:
  bitwarden:
    image: bitwarden/nginx:1.44.1
```

说明：

- `http://bitwarden:80` 中的 `bitwarden` 是 `services` 中的 service 名称，运行时会解析为 service 实际 IP。
- `http://bitwarden.cloud.lazycat.app.bitwarden.lzcapp:80` 写法为 `$service_name.$appid.lzcapp`。
- `lzcos 1.3.x` 后引入应用隔离，应用之间禁止相互访问。无特殊原因时，应用内服务建议直接使用 `service_name`。
- 如果上游服务检查 `Host`，可能需要使用 `xxx.lzcapp` 域名形式，或配置 `upstreams.[].use_backend_host=true`。

### 5.2 file 上游

`file` 路由用于加载静态 HTML 文件，例如纯前端应用：

```yml
application:
  subdomain: pptist
  routes:
    - /=file:///lzcapp/pkg/content/
  file_handler:
    mime:
      - x-lzc-extension/pptist
    actions:
      open: /?file=%u
```

静态资源通常通过 LPK 的 `contentdir` 打包，运行时以只读形式存放在 `/lzcapp/pkg/content/`。

### 5.3 exec 上游

`exec://$port,$exec_file_path` 由两部分组成：

- `$port`：最终提供服务的端口，隐含 host 为 `127.0.0.1`。
- `$exec_file_path`：可执行文件路径，可以是脚本或 ELF 文件。

启动时系统会执行 `$exec_file_path`，并假设服务运行在 `http://127.0.0.1:$port`。系统本身不会检测该服务是否真的由该文件启动。

示例：

```yml
application:
  image: registry.lazycat.cloud/lzc/lzc-files:v0.1.47
  subdomain: file
  routes:
    - /api/=exec://3001,/lzcapp/pkg/content/backend
    - /files/=http://127.0.0.1:3001/files/
    - /=file:///lzcapp/pkg/content/dist
```

### 5.4 UpstreamConfig

`lzcos v1.3.8+` 可使用 `applications.upstreams` 配置更细致路由行为：

```yml
subdomain: debug

routes:
  - /=http://app1.org.snyh.debug.whoami.lzcapp:80

upstreams:
  - location: /search
    backend: https://baidu.com/
    use_backend_host: true
    disable_auto_health_checking: true
    remove_this_request_headers:
      - origin
      - Referer
    disable_url_raw_path: true

  - location: /other
    backend: https://app2.snyh.debug.lzcapp:4443
    disable_backend_ssl_verify: true

  - location: /
    domain_prefix: config
    backend: http://config.snyh.debug.lzcapp:80

  - location: /api
    backend: http://127.0.0.1:3001/
    backend_launch_command: /lzcapp/pkg/content/my-super-backend -listen :3001
```

## 6. LPK 格式与文件边界

LPK v2 规则以前提 `lzcos v1.5.0+` 为基础。如需构建 LPK v2，应配合 `lzc-cli v2.0.0+`。

典型 LPK 顶层结构：

```text
.
├── manifest.yml
├── package.yml
├── content.tar | content.tar.gz
├── images/
├── images.lock
├── exports/
└── META/
```

文件职责：

- `manifest.yml`：应用运行结构定义。
- `package.yml`：静态包元数据。
- `content.tar` / `content.tar.gz`：可选内容归档；未配置 `contentdir` 时可以不存在。
- `images/` 与 `images.lock`：LPK v2 embed image 数据。
- `exports/`：可选，LPK 导出的静态资源。
- `META/`：归档元信息。

`manifest.yml` 只描述运行结构，例如：

- `usage`
- `application`
- `services`
- `ext_config`

LPK v2 中，以下静态字段不应再放在 `manifest.yml`：

- `package`
- `version`
- `name`
- `description`
- `locales`
- `author`
- `license`
- `homepage`
- `min_os_version`
- `unsupported_platforms`

`package.yml` 示例：

```yml
package: cloud.lazycat.app.demo
version: 0.0.1
name: Demo
description: Demo app
locales:
  en:
    name: Demo
    description: Demo app
author: demo
license: MIT
homepage: https://example.com
min_os_version: 1.0.0
unsupported_platforms:
  - ios
import_resources:
  - kind: skills
```

规则：

- LPK v2 tar 格式必须包含 `package.yml`。
- LPK v1 zip 格式仍兼容旧布局，允许静态字段继续留在 `manifest.yml`。
- 新项目应统一把静态字段放在 `package.yml`。
- `import_resources` 用于声明当前 LPK 希望在运行时导入的资源类型。

`content.tar` / `content.tar.gz` 规则：

- 只有配置了非空 `contentdir`，或构建 hook 实际写入内容时，才会生成内容归档。
- image-only release 可以不带 `content.tar*`。
- tar-based LPK v2 若同时存在 `images/`，内容归档可能被压缩为 `content.tar.gz`。

`images/` 与 `images.lock` 规则：

- `images/` 保存 OCI layout。
- `images.lock` 记录 alias、最终 digest 和 upstream 信息。
- `manifest.yml` 中的 `embed:<alias>@sha256:<digest>` 必须与 `images.lock` 对应记录一致。

LPK 资源导出路径：

```text
exports/<kind>/<resource-id>/...
```

运行时导入目录：

```text
/lzcapp/run/resources/<kind>/<package-id>/<resource-id>/...
```

对应 digest 元数据目录：

```text
/lzcapp/run/resources/.digest/<kind>/...
```

## 7. 内嵌镜像与持久化边界

持久化边界：

- `/lzcapp/var` 会在重启后保留。
- 运行态目录，例如 `/home/lazycat`，手工改动会丢失，需要通过镜像或启动脚本固化。

文稿目录边界：

- `/lzcapp/run/mnt/home` 是旧路径，保持原有用户文稿语义。从 `v1.7.0` 开始，只有管理员明确授权后应用才能访问。
- `/lzcapp/documents` 是当前应用自己的应用文稿根目录，实际数据路径为 `/lzcapp/documents/<uid>`。
- 只有在 `package.yml.permissions` 中声明 `document.private` 后，系统才会提供 `/lzcapp/documents`。
- 声明 `document.private` 不会获得其他应用的应用文稿目录。

LPK v2 embed image 用于基于上游镜像做小改动并发布。目标系统建议 `lzcos 1.5.0+`，并确保已能执行：

```bash
lzc-cli project deploy
```

## 8. 发布应用

发布前需要：

- 注册社区账号。
- 访问开发者中心，根据界面引导提交开发者审核。
- 提交应用到商店审核前阅读应用上架审核指南。

通过命令行提交审核要求 `lzc-cli 1.2.54+`：

```bash
lzc-cli project build
lzc-cli appstore publish ./your-app.lpk
```

如果 LPK 中引用外部镜像，最终上传商店前需要将镜像推送到官方 registry，并手动调整 `manifest.yml` 中的相关引用，否则审核人员可能无法安装应用。

推送镜像：

```bash
lzc-cli appstore copy-image <公网可以访问的镜像名称>
```

示例：

```bash
lzc-cli appstore copy-image alpine:3.18
```

完成后会输出类似：

```text
registry.lazycat.cloud/<community-username>/<镜像名称>:<hash版本>
```

官方 registry 限制：

- 为保证 LPK 引用镜像稳定，生成镜像 tag 会替换成 `IMAGE_ID`。
- 每次执行 `copy-image`，服务端都会强制执行一次 `docker pull`。
- 被上传的镜像必须公网可访问；仅本地存在的镜像无法被 `copy-image`。
- 被上传镜像必须被至少一个商店应用引用，仓库会定期垃圾回收。
- `registry.lazycat.cloud` 仅供微服内部使用，在微服外部使用会有限速。

## 9. 移植 Docker 应用

移植已有 Docker 应用时，重点把 Docker 参数转换为 `lzc-manifest.yml`。

Docker 参数对应关系：

- `--env` 对应 `services.<name>.environment`。
- `--publish` 中的 HTTP/HTTPS 服务建议转成 `application.routes`。
- 非 HTTP 服务，例如 SSH 22 端口，使用 `application.ingress`。
- `--volume` 对应 `services.<name>.binds`。
- Docker 镜像名对应 `services.<name>.image`。

`binds` 左侧路径必须以 `/lzcapp` 开头，通常使用 `/lzcapp/var` 或 `/lzcapp/cache`。

GitLab service 示例：

```yml
services:
  gitlab:
    image: gitlab/gitlab-ee:17.2.8-patch7
    binds:
      - /lzcapp/var/config:/etc/gitlab
      - /lzcapp/var/logs:/var/log/gitlab
      - /lzcapp/var/data:/var/opt/gitlab
    environment:
      - GITLAB_OMNIBUS_CONFIG=external_url 'http://gitlab.example.com'
```

完整配置示例：

```yml
# package.yml
package: cloud.lazycat.app.gitlab
version: 17.2.8-patch1
name: GitLab
```

```yml
# lzc-manifest.yml
lzc-sdk-version: '0.1'
application:
  routes:
    - /=http://gitlab.cloud.lazycat.app.gitlab.lzcapp:80
  subdomain: gitlab
  ingress:
    - protocol: tcp
      port: 22
      service: gitlab

services:
  gitlab:
    image: gitlab/gitlab-ee:17.2.8
    binds:
      - /lzcapp/var/config:/etc/gitlab
      - /lzcapp/var/logs:/var/log/gitlab
      - /lzcapp/var/data:/var/opt/gitlab
    environment:
      - GITLAB_OMNIBUS_CONFIG=external_url 'http://gitlab.${LAZYCAT_BOX_DOMAIN}'; gitlab_rails['lfs_enabled'] = true;
```

补齐 `lzc-build.yml`：

```yml
pkgout: ./
icon: ./icon.png
```

目录结构：

```text
.
├── lzc-build.yml
├── lzc-manifest.yml
├── package.yml
└── icon.png
```

构建并安装：

```bash
lzc-cli project build
lzc-cli lpk install ./cloud.lazycat.app.gitlab-v17.2.8-patch1.lpk
```

从 `docker-compose.yml` 转换时，重点关注：

- `environment`
- `ports`
- `volumes`
- `image`

转换规则与 Docker 命令一致。

常见问题：

- 镜像拉不下来：自用场景可考虑使用 `lzc-build.yml` 的 `images` 构建机制将镜像内嵌到 LPK；公用场景需要自行建立 registry 或使用官方 registry 流程。
- 可参考官方移植开源仓库：`https://gitee.com/lazycatcloud/appdb`

## 10. 数据库服务

MySQL service 示例：

```yml
services:
  mysql:
    image: registry.lazycat.cloud/mysql
    binds:
      - /lzcapp/var/mysql:/var/lib/mysql
    environment:
      - MYSQL_ROOT_PASSWORD=LAZYCAT
      - MYSQL_DATABASE=LAZYCAT
      - MYSQL_USER=LAZYCAT
      - MYSQL_PASSWORD=LAZYCAT
```

字段说明：

- `mysql`：数据库 Docker 服务名。
- `image`：MySQL Docker 镜像地址。
- `binds`：将容器内 `/var/lib/mysql` 持久化到 `/lzcapp/var/mysql`。
- `environment`：MySQL 启动环境变量。

连接方式：

```text
mysql.<package>.lzcapp:3306
```

例如应用 `package` 为 `cloud.lazycat.app.todolistpy`，代码中访问：

```text
mysql.cloud.lazycat.app.todolistpy.lzcapp:3306
```

PostgreSQL 示例：

```yml
services:
  postgresql:
    image: registry.lazycat.cloud/postgres:18.1
    environment:
      - POSTGRES_USER=lazycat
      - POSTGRES_PASSWORD=lazycat
      - POSTGRES_DB=lazycat
    binds:
      - /lzcapp/var/pgdata:/var/lib/postgresql
```

Redis 原始材料被截断，只能确认示例开头：

```yml
services:
  redis:
    image: registry.lazycat.cloud/redis:8.0
```

## 11. 平台支持

微服官方应用天然支持 Linux、Windows、macOS、Android、iOS、鸿蒙等平台。如应用不支持某些平台，可在配置中加入 `unsupported_platforms`。

示例：

```yml
unsupported_platforms:
  - ios
```

参数表：

| 参数 | 平台 |
| --- | --- |
| `ios` | 不支持 iOS 和 iPad 移动端 |
| `android` | 不支持 Android 移动端 |
| `linux` | 不支持 Linux 桌面端 |
| `windows` | 不支持 Windows 桌面端 |
| `macos` | 不支持 macOS 桌面端 |
| `tvos` | 不支持懒猫智慧屏平台端，要求系统 1.0.18 以上 |

## 12. 部署参数

`lzc-deploy-params.yml` 用于定义安装时参数。

顶层结构：

| 字段名 | 类型 | 描述 |
| --- | --- | --- |
| `params` | `[]DeployParam` | 开发者定义的部署参数列表 |
| `locales` | `map` | 国际化相关 |

`DeployParam` 字段：

| 字段名 | 类型 | 描述 |
| --- | --- | --- |
| `id` | `string` | 应用内唯一 ID，供国际化和 `manifest.yml` 引用 |
| `type` | `string` | 字段类型，目前支持 `bool`、`lzc_uid`、`string`、`secret` |
| `name` | `string` | 字段渲染时名称，支持国际化 |
| `description` | `string` | 字段渲染时详细介绍，支持国际化 |
| `optional` | `bool` | 是否可选；若所有字段均可选，会直接跳过部署界面 |
| `default_value` | `string` | 默认值，支持 `$random(len=5)` 生成随机串，要求 `lzcos 1.5.0+` |
| `hidden` | `bool` | 字段生效，但不在界面渲染 |

## 13. 开发与发布检查清单

开发前：

- 已安装 Node.js 18+。
- 已安装 `@lazycatcloud/lzc-cli`。
- `lzc-cli box default` 指向目标微服。
- 能执行 `lzc-cli project --help`。

开发态：

- `lzc-cli project deploy` 完成。
- `lzc-cli project info` 显示应用已运行。
- Web 或客户端能打开入口。
- 前端 dev server 启动后能热更新。
- 可通过 `lzc-cli project log -f` 排查。

后端与路由：

- `exec://` 用于启动进程并转发流量。
- `http://` 用于转发到已存在 service。
- 更具体路由放在更宽泛路由前。
- 必要的公开路径使用 `application.public_path` 最小放开。
- 持久化数据写入 `/lzcapp/var`。

发布前：

- 静态元数据放在 `package.yml`。
- 运行结构放在 `lzc-manifest.yml`。
- 镜像可被目标微服拉取，商店发布前按需推送到官方 registry。
- 执行 `lzc-cli project build`。
- 使用 `lzc-cli lpk info <file.lpk>` 检查包信息。
- 使用 `lzc-cli appstore publish ./your-app.lpk` 提交审核。
